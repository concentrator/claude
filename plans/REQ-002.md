# REQ-002: Restructure the planning directory layout

## Current state

Everything planning-related lives flat in `.claude/plans/`: foundational
indexes (`roadmap.md`, `tasks.md`), per-initiative `REQ-XXX.md`, branch
plans named by bare slug, `.findings.md` companions, batch manifests
(`B-XXX.md`), and release plans. One completed agent batch in
wallarm-api-js produced 30 files in a single directory. Pain points:

- Branch plan filename = bare slug → unordered listing, no visible
  file→task mapping, hard to follow.
- Indexes consulted on every planning action are buried among dozens of
  work artifacts.
- No grouping: one initiative's 17 sweep plans interleave with
  everything else. Checkpoint validation (reviewing new plans and
  findings) is needlessly slow, and the next batch makes it worse.

## Desired state

```
.claude/
├── REQUIREMENTS.md             # uppercase per Amendment 1
├── DESIGN.md
├── MAINTENANCE.md
├── ROADMAP.md                  # moved up — navigation index
├── TASKS.md                    # moved up — navigation index
└── plans/
    ├── REQ-XXX.md              # per-initiative requirements (root)
    ├── release-vX.Y.Z.md       # release plans (+ archive/ as today)
    ├── batches/
    │   └── B-XXX.md            # manifests referencing T-XXX ids
    └── R-XXX-<slug>/           # one dir per roadmap entry
        ├── T-XXX-<slug>.md           # branch plan
        └── T-XXX-<slug>.findings.md  # adjacent findings
```

- `roadmap.md` / `tasks.md` sit beside `requirements.md`/`design.md` as
  steering docs; `plans/` holds only units of planned work.
- Branch plans prefixed with their task id: natural sort order, instant
  file→task mapping, findings adjacent. Branch names stay
  `<prefix>/<slug>` — git refs carry no id.
- Roadmap-entry dirs named id + short slug (`R-002-route-sweeps/`),
  never bare `R-XXX/`.
- Batches stay manifests in `plans/batches/` — batch membership is
  transient execution grouping; plans are never filed by batch, so
  re-batching moves no files.
- In the self-hosting repo (`~/.claude`), "`.claude/`" is the repo
  root: `roadmap.md`/`tasks.md` move to the root, `plans/` follows the
  same internal layout.

## Invariants

- ID formats, entry formats, and the four-level hierarchy are
  unchanged — this moves files, not semantics.
- Referential integrity rules unchanged (child requires open parent).
- Plan/finding file *content* is not rewritten beyond path references.
- Archival policy unchanged (closed items stay in place, marked `[x]`;
  release-plan archive exception remains).
- Branch naming, commit rules, agentic rails unchanged.
- Permission templates keep working (`.claude/plans/**` already covers
  subdirectories).

- `rules/project-layout.md` — canonical tree.

## Amendment 1 (2026-06-11): uppercase foundational docs

Rename the five foundational docs to uppercase, matching the
root-meta-document convention (`README.md`, `CHANGELOG.md`,
`CLAUDE.md`): `requirements.md → REQUIREMENTS.md`,
`design.md → DESIGN.md`, `maintenance.md → MAINTENANCE.md`,
`roadmap.md → ROADMAP.md`, `tasks.md → TASKS.md`.

Rationale: ecosystem signal for steering docs, sorts first in
listings, removes the unprincipled case split at `.claude/` root.
Planning artifacts inside `plans/` (`REQ-XXX.md`, plan files, batches,
release plans) keep their existing casing — they are work items, not
root meta-docs.

Scope: rules + skills references, this repo's files, adopter projects.
Sequencing: define + self-rename before T-003 so wallarm-api-js
migrates and renames in a single pass (T-003 plan to be extended
accordingly). Case-only renames must use `git mv` (macOS
case-insensitive FS).
