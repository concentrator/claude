# REQ-001: Parallel batch execution for DEV auto mode

## Motivation

`/dev auto` (engine: `delegating-to-agents`) runs a batch strictly
serially: branches in batch order, one implementer subagent per commit,
each gated by a spec-review and (at close) a code-review agent. For a
batch of branches that ship to **independent components**, this wastes
wall-clock — the slow, isolated work (implement, live-probe, test, lint,
review) of branch B sits idle while branch A finishes. Observed: a
multi-branch refactor batch takes far longer than the underlying work
warrants because every model round-trip is sequential.

## Goals

- Run the **expensive, isolated phase** of multiple branches concurrently
  — implement + probe + test + lint + spec-review + code-review — each in
  its own git worktree branched from the default branch.
- Keep **integration serial**: merges to the local default branch happen
  one at a time at the checkpoint, with the green-gate re-run after each,
  and conflicts on shared files resolved deliberately.
- Let a batch declare its execution mode at plan stage: `serial`
  (today's behavior, default) or `parallel` with a concurrency cap.
- Add a plan-stage **safety gate** that verifies parallel members are
  actually independent (disjoint file-touch sets and no contended shared
  resources), rather than trusting "different components".
- Preserve every existing safety rail: `pre-B-XXX` rollback tag,
  no-commit-on-red, never-push, findings/triage deferred to checkpoint.

## Non-goals

- Parallel *merging* to the default branch (git serializes; integration
  stays one-at-a-time).
- Removing or weakening the green-gate or the linear local-merge model.
- Cross-machine / distributed execution.
- Auto-parallelizing batches that weren't reviewed and stamped for it.

## User experience

- **Batch schema** (`B-XXX.md`) gains a header:
  `mode: serial | parallel` (default `serial`) and, when parallel,
  `concurrency: N` (e.g. 3).
- **Scheduling** respects `depends-on` as a DAG: only members with all
  in-batch dependencies merged are eligible to start; eligible members
  run up to `concurrency` at a time in worktrees. A batch with no deps
  and disjoint files runs as one wave.
- **Worktree isolation**: each concurrent agent works in a worktree off
  the default branch; the main checkout is never the work surface for a
  parallel member.
- **Integration phase**: as branches complete review, they queue for a
  serial merge at the checkpoint — merge, re-run green-gate, resolve
  shared-file conflicts (see edge cases), proceed to the next.
- **Halt isolation**: one member going BLOCKED / failing its gate does
  not abort in-flight siblings; their outcomes are collected and reported
  together at the checkpoint.
- **Edge cases**:
  - Append-only / registration shared files that independent components
    still both touch (in a route SDK: the central class registry,
    CHANGELOG, public README, and shared plan files like `TASKS.md` /
    `ROADMAP.md` checkboxes) collide on merge. These route to the serial
    integration phase rather than being edited in parallel worktrees.
  - Shared **mutable external resources** (e.g. a live test tenant that a
    probe could create/mutate against): not part of currently planned
    work, but a parallel member that mutates shared external state races
    with siblings. The readiness gate must screen for this and force such
    members serial when present.

## Acceptance criteria

- [ ] A batch can declare `mode: parallel` + `concurrency: N`; absent →
      serial (unchanged behavior).
- [ ] Parallel members each execute in an isolated git worktree off the
      default branch; the main checkout is untouched until integration.
- [ ] Up to `N` members run concurrently; `depends-on` is honored as a
      DAG (a member starts only once its in-batch deps are merged).
- [ ] Merges to the local default branch are serial, each followed by the
      project's test + lint gate; a red gate halts integration.
- [ ] A plan-stage readiness gate rejects `parallel` unless members have
      disjoint file-touch sets (modulo a declared shared-file set routed
      to serial integration) and no contended mutable external resource.
- [ ] A single member's halt/failure does not abort siblings; all
      outcomes surface at the checkpoint.
- [ ] `pre-B-XXX` rollback restores the pre-batch state regardless of how
      many members completed.

## Constraints

- Git worktree isolation is mandatory for any concurrent member — a
  shared checkout corrupts the index/working tree.
- Concurrency must be bounded (explicit cap) to respect machine and
  external-API limits.
- The existing serial path must remain the default and stay byte-for-byte
  behaviorally unchanged when `mode` is absent or `serial`.

## Open questions

- How are per-branch file-touch sets declared/derived at plan stage —
  authored in the branch plan, or inferred from the commit items?
- How are shared append-only files handled at integration — automatic
  conflict resolution, a dedicated integration commit, or per-file merge
  drivers?
- Default `concurrency` when `parallel` is declared without a value?
- Partial-success policy at checkpoint: integrate the green members and
  report the rest, or hold all pending user decision?
