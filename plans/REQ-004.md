# REQ-004: Requirement lifecycle closure

## Motivation

Every planning level has defined closure except the top. Branch-plan
checkboxes close at commit, tasks at merge, roadmap items when child
tasks close — but REQ files have no closure semantics: no point in the
flow verifies acceptance criteria, no completion marker exists, and
"open REQ" (which referential integrity depends on) is undefined.
Observed in practice: REQ-002's criteria were ticked only ad-hoc;
REQ-003's would have drifted unchecked after R-002 closed.

## Goals

- Define REQ open/closed: frontmatter `status:` field — open while
  any acceptance criterion is `[ ]`; `status: done YYYY-MM-DD` only
  when all are `[x]` with evidence.
- Add the verification trigger: when the last open R under a REQ
  closes (finishing-a-branch § 4), walk the REQ's acceptance criteria —
  tick what is verified (with one-line evidence), leave what is not,
  and report the pending set explicitly.
- Criteria that need future events (live runs, releases) keep the REQ
  open with the pending criteria visible; the next relevant event
  re-triggers verification (e.g. the first `/dev auto` batch validates
  REQ-003's run-dependent criteria at its checkpoint).
- Update planning.md: REQ lifecycle definition + frontmatter field;
  referential-integrity wording rests on the defined "open".

## Non-goals

- Retroactive rewriting of closed REQs' history.
- Auto-closing REQs without evidence per criterion.

## Acceptance criteria

- [ ] planning.md defines REQ open/closed and the `status:` field;
      REQ template includes it.
- [ ] finishing-a-branch § 4 walks parent-REQ acceptance criteria when
      the last open child R closes; pending criteria are reported, not
      silently skipped.
- [ ] delegating-to-agents checkpoint re-triggers verification for
      run-dependent criteria of REQs awaiting validation.
- [ ] REQ-002 and REQ-003 backfilled: criteria verified/ticked with
      evidence, `status:` set per the new rule.

## Constraints

- Closure markers must not break existing `approved:` semantics.
- finishing-a-branch and delegating-to-agents word caps hold.

## Open questions

- Should `/dev plan REQ` listing (`/dev` bare routing) surface REQs by
  status to make drift visible?
