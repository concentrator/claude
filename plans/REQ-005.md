# REQ-005: Flatten the requirement level into roadmap entries

Supersedes REQ-004 (requirement lifecycle closure) — its goals are
absorbed by the single-closure-point design below. The last REQ-XXX
born under the old scheme.

## Current state

Four-level chain `REQ-XXX → R-XXX → T-XXX → branch plan` with two
independent id spaces at the top. In practice every REQ has spawned
exactly one R (REQ-002→R-001, REQ-003→R-002, wallarm REQ-001→R-001) —
pure indirection: three lookups across three locations to trace one
feature, REQ files placed at `plans/` root physically apart from the
R-dirs that implement them, and no closure semantics for REQs (the
REQ-004 gap). The `/dev plan` surface needs three targets (`REQ`,
`REQ-XXX`, `roadmap`) for what is conceptually one act. Indexes
(`ROADMAP.md`/`TASKS.md`) sit at `.claude/` root, apart from the
content they index — the burial problem that justified moving them up
was cured by the R-dirs themselves (REQ-002).

## Desired state

```
.claude/
├── REQUIREMENTS.md             # foundational — unchanged
├── DESIGN.md
├── MAINTENANCE.md
└── plans/
    ├── ROADMAP.md              # initiative index — returns here
    ├── TASKS.md                # task index — returns here
    ├── batches/
    ├── archive/
    └── R-XXX-<slug>/           # one dir per initiative
        ├── requirements.md     # same template, no own id; points to R-XXX
        ├── T-XXX-<slug>.md
        └── T-XXX-<slug>.findings.md
```

- Chain: `R-XXX → T-XXX → branch`. The roadmap entry is the chain
  root; the dir name carries the id.
- New initiative = one act: ROADMAP entry + dir + `requirements.md`
  (`approved: pending`). Dir creation moves from lazy to
  initiative-time. Nothing downstream while pending (gate unchanged,
  relocated).
- Single closure point: an R entry closes only when all child tasks
  are `[x]` AND its `requirements.md` acceptance criteria are verified
  (`status: done YYYY-MM-DD` in its frontmatter, evidence per
  criterion). Run-dependent criteria keep the R open; the relevant
  event (e.g. batch checkpoint) re-triggers verification — absorbs
  REQ-004 in full.
- `/dev plan` surface: `REQ`, `REQ-XXX`, `roadmap` targets merge into
  `R` (new initiative via `brainstorming`) and `R-XXX` (state-routed:
  requirements pending → shape/approve; approved → add tasks).
- Findings promotion: no fitting open R → create an R stub (entry +
  dir + requirements draft) — replaces "promote to REQ-XXX".

## Invariants

- The four artifacts (requirement content, roadmap entry, task,
  branch plan) all still exist — only the id space and placement
  change. Requirement template sections unchanged.
- `approved:` gating semantics unchanged.
- T-XXX/B-XXX id schemes, branch naming, batch flow, rails unchanged.
- Foundational `REQUIREMENTS.md` at `.claude/` root unchanged.
- History is not rewritten: closed REQ-002/REQ-003 (and superseded
  REQ-004) stay at `plans/` root as read-only history.

## Scope

- `rules/planning.md` — hierarchy, id format, integrity rules, REQ
  template relocation, closure rule, directory conventions.
- Skills: `dev` (routing table), `brainstorming` (creates R entry +
  dir + requirements), `writing-plans` (chain resolution),
  `finishing-a-branch` (R closure step incl. criteria verification),
  `delegating-to-agents` + prompts ("REQ acceptance criteria" wording,
  checkpoint re-verification), `starting-a-project`,
  `migrating-to-dev`, `release` (ROADMAP path).
- `CLAUDE.md` § Temporary Files, `README.md`, `DESIGN.md` tree-map,
  `MAINTENANCE.md` targets — paths.
- Index move: `ROADMAP.md`/`TASKS.md` → `plans/` (this repo +
  adopters, one migration pass with the flatten).
- Migrations: REQ-001 (pending) → `requirements.md` of its future
  R-dir on approval; wallarm-api-js REQ-001 + indexes — single pass,
  between batches.

      `R-XXX` routes by requirements state; `REQ`/`roadmap` targets

- [ ] Indexes live at `plans/` root here and in wallarm-api-js; no
