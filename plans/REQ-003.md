# REQ-003: Batch integration branch, enforced review report, push + MR at checkpoint

## Motivation

First completed `/dev auto` batch (wallarm-api-js B-001, 7 branches)
exposed two checkpoint gaps:

- The detailed code-review report required by the skill text arrived as
  a bare "accept findings?" offer — conversational requirements degrade;
  the report must be a written artifact.
- All work was merged to **local** main and stopped there. The batch is
  not done until it is reviewed at full strength, fixed, tested, docs
  updated, **pushed to origin, and an MR created**. Deferring the push
  is the user's explicit opt-out, not the default.

## Goals

- Introduce a batch integration branch type: `batch/B-XXX`, created at
  pre-flight off the default branch; member branches merge into it.
  Local default branch is never touched during a batch run.
- Add a batch-level close phase: full-diff review of `batch/B-XXX` vs
  default by the most capable model (cross-branch interactions —
  invisible to per-branch reviews), fixes applied as batch-branch
  commits, tests + lint re-run, docs coherence pass (CHANGELOG/README
  across all member branches).
- Enforce a written checkpoint report: `.claude/plans/B-XXX.report.md`
  — per branch: commits, test evidence, review findings and their
  resolutions; plus batch-level findings, fixes, prompt-friction
  summary. Presented at checkpoint; accept is invalid without it.
- On accept: push `batch/B-XXX` to origin and create the MR
  (`glab`/`gh`; description sourced from the report). Push may be
  deferred only by explicit user choice at the checkpoint.
- Narrow the push rail: agents never push mid-batch; the checkpoint
  pushes the batch branch only — never the default branch.

## Non-goals

- Auto-merging the MR (review/merge on origin stays with the user).
- Changing per-branch close (spec check, branch-level code review stay).
- Changing manual-mode (`/dev code`) flow — `finishing-a-branch`
  already offers push/MR per branch.

## User experience

- `/dev auto B-XXX` → pre-flight creates `batch/B-XXX` → branches
  execute and merge into it → batch close phase (review, fix, retest,
  docs) → checkpoint shows `B-XXX.report.md` → accept → push + MR link
  printed. Rollback = delete batch branch (`pre-B-XXX` tag stays as
  belt-and-braces).
- Reject and halt flows unchanged except main was never modified, so
  reject no longer needs `git reset --hard` on main.
- Release plans: a batch's tasks close `[x]` when the batch MR merges
  to the default branch (today: at local merge).

## Acceptance criteria

- [x] Pre-flight creates `batch/B-XXX`; member branches merge into it;
      default branch untouched during the run. (B-002: report header
      `batch-branch: batch/B-002`, 4/4 merged into it.)
- [x] Batch close runs the full-diff review on the most capable model,
      applies fixes on the batch branch, re-runs tests + lint, and does
      a docs coherence pass. (B-002 report § Batch review + § Docs
      coherence; caught the Hints clientId cross-branch hazard.)
- [x] `B-XXX.report.md` exists and contains per-branch and batch-level
      sections before the checkpoint offer is made. (B-002 report,
      enforced pre-offer.)
- [x] Accept pushes `batch/B-XXX` to origin and creates an MR; the MR
      description references the report content. (B-002 MR pushed +
      merged.)
- [x] An explicit "defer push" checkpoint choice exists; nothing else
      skips the push. (delegating SKILL checkpoint Accept wording.)
- [x] Agents never push mid-batch; nothing ever pushes the default
      branch (rails wording; carve-out allows only `batch/*`; B-002
      raised zero mid-batch push prompts).
- [x] Task checkboxes close on MR merge, not local merge. (branch-plan
      rails + finishing-a-branch §4 auto-mode gate.)

## Constraints

- Requires VCS-host CLI (`glab`/`gh`) in the project's
  `## Agent toolchain`; missing → checkpoint falls back to push +
  manual MR instruction, never silently skips the push.
- `Bash(git push:*)` deny rules in adopter projects need a carve-out
  for `git push origin batch/B-XXX` at checkpoint (controller-only),
  or checkpoint push runs with explicit user approval of that single
  command.

## Open questions

- Batch branch naming: `batch/B-001` vs including theme slug.
- One MR per batch confirmed, or optionally per-branch MRs from the
  batch branch for large batches?
- Should `B-XXX.report.md` live under the future `plans/batches/`
  (REQ-002 layout) — align if both land.
